Systematic ML Methodology

Validation-first approach for production-ready predictive maintenance

5-Step Pipeline

1Ô∏è‚É£ Data Validation

Goal: Ensure data quality and understand temporal patterns

  • Validate SCADA structure (10-min intervals)
  • Analyze failure timing patterns (7-day vs sudden)
  • Critical insight: status==0 AND power>0.1 for true normal operation
  • Create ¬±30 day exclusion zones around failures

2Ô∏è‚É£ Feature Engineering

Goal: Create meaningful aggregated features from raw sensor data

  • Extract 24-hour windows (144 timestamps)
  • Compute statistics: mean, std, linear trend
  • Generates 3√ó features per sensor (243-2,856 features)
  • Captures both level and variability

3Ô∏è‚É£ Feature Selection

Goal: Select discriminative, non-redundant features

  • Cohen's d for effect size (threshold: 0.8-1.0)
  • Iterative correlation removal (r < 0.9)
  • Discovered massive redundancy (214 pairs r>0.9)
  • Final: 15-50 features per farm

4Ô∏è‚É£ Model Training

Goal: Train interpretable model with proper validation

  • Random Forest (n=100, max_depth=5, balanced)
  • Leave-One-Out CV (appropriate for n=6-27 failures)
  • Chose interpretability over performance
  • Per-event probability tracking

5Ô∏è‚É£ Transferability Validation

Goal: Verify methodology works across different environments

  • Farm C independent validation (67% matches Farm A)
  • Different location, turbine model, sensor count
  • Demonstrates methodology generalizability
  • Production-ready confidence

Critical Methodology Decisions

‚úì

Random Forest > Deep Learning

Rationale: Small dataset (n=6-27 failures) makes deep learning impractical. Random Forest provides interpretability critical for operator trust while maintaining strong performance.

Alternative considered: LSTM (rejected - requires 100s of failures)

üìä

Window Statistics > Raw Time Series

Rationale: Aggregating 24h windows into mean/std/trend reduces dimensionality while capturing essential patterns. Enables consistent feature space across 81-952 sensors.

Alternative considered: CNN on raw sequences (rejected - too many sensors)

üéØ

Cohen's d > Univariate Tests

Rationale: Cohen's d measures practical significance (effect size), not just statistical significance. More appropriate for small samples where p-values can be misleading.

Threshold: d > 0.8-1.0 (large effect size)

üîÑ

Leave-One-Out CV > K-Fold

Rationale: With only 6-27 failures, standard k-fold would have <2 failures per fold. LOO maximizes training data (n-1) while providing unbiased estimates.

Trade-off: Computationally expensive but necessary

‚öñÔ∏è

Precision > Recall Trade-off

Rationale: False alarms cost $5K-$10K (unnecessary inspection) and erode operator trust. Optimized for 95%+ precision while maximizing recall within this constraint.

Result: 100% validation precision achieved

üè≠

¬±30 Day Exclusion Zones

Rationale: Prevents data leakage from pre-failure degradation (7-day pattern) or post-failure anomalies. Ensures true "normal" baseline windows.

Validation: Tested 15, 30, 45 days (30 optimal)

Unified Multi-Farm Model

The Problem

Challenge: Farm B has only 6 failures (below minimum 10-12 needed for reliable ML)

Result: 33% recall - unreliable, cannot deploy

Question: Can we leverage data from other farms to improve Farm B?

The Solution: Semantic Mapping

Approach: Map farm-specific sensors to unified physical categories

Example:

  • Farm A: sensor_48 ‚Üí rotor_mechanical
  • Farm B: sensor_20 ‚Üí rotor_mechanical
  • Farm C: sensor_100 ‚Üí rotor_mechanical

Result: 16 shared categories across all farms

Category Merging Process

Initial: 50 granular categories (88% missing data)

Merges Applied: 38 documented merges

Examples:

  • electrical_general + electrical_grid + electrical_internal ‚Üí electrical_grid
  • gearbox_bearings + gearbox_main_shaft ‚Üí gearbox_mechanical
  • generator_current + generator_speed ‚Üí generator_electrical

Final: 16 categories (25% missing)

Results & Trade-offs

Farm B Improvement: 33% ‚Üí 50% (+17%)

Training data: 6 ‚Üí 45 failures (7.5√ó increase)

Trade-off: Farms A & C degrade -7 to -9%

Root cause: Aggregating sensors within categories loses granular information

Conclusion: Unified best for limited-data; farm-specific preferred when possible

Lessons Learned

üí° Data Quality > Model Complexity

Spent 40% of time on data validation and cleaning. Critical insight about status filtering (status==0 AND power>0.1) discovered through EDA, dramatically improved performance.

üìä Feature Engineering is Key

Window aggregation (mean, std, trend) captures 90% of signal. Adding max/min provided minimal gain. Simple statistics on domain-informed features > complex architectures.

‚öñÔ∏è Multi-Domain Learning Trade-offs

Unified approach improves limited-data scenarios (+17%) but sacrifices peak performance on well-instrumented farms (-7 to -9%). Generalizability vs. optimization is fundamental.

üéØ Validation Strategy Matters

Independent validation (Farm C) provided confidence in methodology transferability. LOO CV appropriate for small n. Real-world validation > perfect metrics on training set.

üìù Documentation Pays Off

Systematic documentation of every decision, trade-off, and alternative considered creates portfolio value. 30,000+ words across all methodology documents demonstrates rigorous thinking.

üîß Interpretability for Production

Operator trust requires understanding WHY the model predicts failure. Random Forest feature importance + physical category names make predictions explainable and actionable.
